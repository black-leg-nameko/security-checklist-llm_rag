# LLM/RAG セキュリティチェックリスト

## 1. 入力検証・サニタイゼーション

### 1.1 プロンプトインジェクション対策
- [ ] プロンプトインジェクション攻撃の検出機能が実装されているか
- [ ] システムプロンプトとユーザー入力を適切に分離しているか
- [ ] プロンプトテンプレートの検証機能があるか
- [ ] 特殊な文字列（例: `</s>`, `[INST]`, `<<SYS>>`）のフィルタリングがあるか

### 1.2 入力長制限
- [ ] 入力テキストの最大長が設定されているか
- [ ] トークン数の上限が設定されているか
- [ ] 長すぎる入力に対する適切なエラーハンドリングがあるか

### 1.3 入力形式の検証
- [ ] 入力データの型チェックが行われているか
- [ ] 不正な文字エンコーディングの処理があるか
- [ ] ファイルアップロードがある場合、ファイルタイプの検証があるか

## 2. 出力検証・フィルタリング

### 2.1 不適切なコンテンツの検出
- [ ] 有害なコンテンツのフィルタリング機能があるか
- [ ] 出力内容の検証プロセスがあるか
- [ ] コンテンツモデレーションAPIが統合されているか

### 2.2 機密情報の漏洩防止
- [ ] 出力に機密情報（APIキー、パスワード、個人情報など）が含まれていないか検証しているか
- [ ] PII（個人識別情報）の検出・マスキング機能があるか
- [ ] 出力ログに機密情報が記録されないようになっているか

### 2.3 出力形式の検証
- [ ] 出力が期待される形式（JSON、XMLなど）に準拠しているか検証しているか
- [ ] 出力の構造が破損していないかチェックしているか

## 3. 認証・認可

### 3.1 APIキー管理
- [ ] APIキーが環境変数やシークレット管理サービスで管理されているか
- [ ] APIキーがコードにハードコードされていないか
- [ ] APIキーのローテーション機能があるか
- [ ] 無効なAPIキーに対する適切なエラーメッセージがあるか

### 3.2 アクセス制御
- [ ] ユーザー認証機能が実装されているか
- [ ] ロールベースのアクセス制御（RBAC）があるか
- [ ] リソースへのアクセス権限が適切に管理されているか

### 3.3 レート制限
- [ ] API呼び出しのレート制限が実装されているか
- [ ] DDoS攻撃に対する保護機能があるか
- [ ] レート制限の設定が適切か

## 4. データ保護

### 4.1 暗号化
- [ ] 機密データが保存時に暗号化されているか（暗号化保存）
- [ ] 通信がHTTPS/TLSで保護されているか（転送時暗号化）
- [ ] 暗号化アルゴリズムが適切か（例: AES-256）

### 4.2 データ保持ポリシー
- [ ] データの保持期間が定義されているか
- [ ] 不要なデータの自動削除機能があるか
- [ ] データ削除のログが記録されているか

### 4.3 プライバシー保護
- [ ] GDPR、個人情報保護法などの法規制に準拠しているか
- [ ] ユーザーデータの匿名化・仮名化が行われているか
- [ ] データ処理の同意取得プロセスがあるか

## 5. モデルセキュリティ

### 5.1 モデルの改ざん検知
- [ ] モデルファイルの整合性チェック機能があるか
- [ ] モデルのハッシュ値検証が行われているか
- [ ] モデルのバージョン管理が適切に行われているか

### 5.2 依存関係の管理
- [ ] 使用しているライブラリの脆弱性スキャンが行われているか
- [ ] 依存パッケージのバージョンが固定されているか（requirements.txt, Pipfile.lockなど）
- [ ] 定期的なセキュリティアップデートが行われているか

### 5.3 モデル設定
- [ ] モデルの設定ファイルに機密情報が含まれていないか
- [ ] デフォルト設定が安全か（例: デバッグモードが本番で無効になっているか）

## 6. RAG固有のセキュリティ

### 6.1 ベクトルDBのアクセス制御
- [ ] ベクトルデータベースへのアクセスが適切に制限されているか
- [ ] インデックスのアクセス権限が適切に設定されているか
- [ ] ベクトルDBの認証情報が安全に管理されているか

### 6.2 ドキュメントの信頼性検証
- [ ] インデックスに追加されるドキュメントの出所が検証されているか
- [ ] 悪意のあるドキュメントの検出機能があるか
- [ ] ドキュメントの改ざん検知機能があるか

### 6.3 情報漏洩リスク
- [ ] 検索結果に機密情報が含まれていないかチェックしているか
- [ ] コンテキストウィンドウに含まれる情報が適切にフィルタリングされているか
- [ ] RAGパイプラインの各ステップでログが記録されているか

## 7. ログ・監視

### 7.1 ログ管理
- [ ] セキュリティ関連のイベントがログに記録されているか
- [ ] ログに機密情報が含まれていないか
- [ ] ログの保持期間とアーカイブポリシーが定義されているか

### 7.2 監視・アラート
- [ ] 異常なアクセスパターンの検出機能があるか
- [ ] セキュリティインシデントに対するアラート機能があるか
- [ ] パフォーマンス監視が行われているか

## 8. エラーハンドリング

### 8.1 エラーメッセージ
- [ ] エラーメッセージに機密情報が含まれていないか
- [ ] スタックトレースが本番環境で公開されていないか
- [ ] エラーメッセージが適切に一般化されているか

### 8.2 例外処理
- [ ] すべての例外が適切にキャッチされているか
- [ ] 例外処理がセキュリティホールを生み出していないか

## 9. インフラストラクチャ

### 9.1 ネットワークセキュリティ
- [ ] ファイアウォールルールが適切に設定されているか
- [ ] 不要なポートが閉じられているか
- [ ] VPCやプライベートネットワークが適切に設定されているか

### 9.2 コンテナ・デプロイメント
- [ ] コンテナイメージに脆弱性がないか
- [ ] 最小権限の原則に従ってデプロイされているか
- [ ] シークレットが適切に管理されているか（例: Kubernetes Secrets）

## 10. コンプライアンス・ドキュメント

### 10.1 セキュリティドキュメント
- [ ] セキュリティポリシーが文書化されているか
- [ ] インシデント対応計画が策定されているか
- [ ] セキュリティ監査の記録が残されているか

### 10.2 法的コンプライアンス
- [ ] 関連する法規制（GDPR、個人情報保護法など）に準拠しているか
- [ ] 利用規約とプライバシーポリシーが適切に設定されているか

---

## チェックリストの使い方

1. 各項目を順番に確認し、該当する場合はチェックボックスにチェックを入れます
2. 該当しない項目は「N/A」とマークします
3. 問題が見つかった場合は、対応計画を記録します
4. 定期的にこのチェックリストを再確認します

## 備考

- このチェックリストは包括的ですが、プロジェクト固有の要件に応じてカスタマイズしてください
- セキュリティは継続的なプロセスです。定期的な見直しを推奨します
- 専門家によるセキュリティ監査も検討してください
